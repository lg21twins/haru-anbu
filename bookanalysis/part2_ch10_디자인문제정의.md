# 2부 10장: 디자인 문제 정의

## 핵심 요약

디자인 문제 정의는 더블 다이아몬드 프로세스의 첫 번째 다이아몬드(발견→정의)를 마무리하는 단계이다. 리서치를 통해 수집한 데이터에서 사용자의 진짜 니즈를 파악하고, 무엇을 디자인해야 할지 명확히 정의하는 과정이다.

## 1. 디자인 문제 정의의 목적

### 핵심 개념
- 요구 사항과 기능은 같지 않다 — 사람들이 원하는 기대는 결국에는 **제품의 기능으로 구현**되어야 한다
- 그렇지만 요구 사항을 기능으로 바꾸는 것은 **사전적인 변환이 아닌 기준과 관점에 따라 달라지는 해석의 과정**
- 요구 사항이 명확하지 않으면 페인포인트가 잘 드러나지 않을 수 있음

### 좋은 디자인의 조건
- 좋은 디자인을 하려면 무엇보다 **문제를 잘 정의**해야 한다
- 디자인 문제의 핵심은 **근본적인 니즈를 찾아내는 것**
- 근본적 니즈와 잠재적 니즈를 **모두 다루어야** 디자인적 해결책으로 이어짐
- 더블 다이아몬드 모델의 첫 번째 다이아몬드를 통해 **문제 정의**에 이르고, 두 번째 다이아몬드에서 **문제 해결**로 이어짐

## 2. 요구 사항에 영향을 주는 요인

### 영향 요인들
- **기대 수준, 성능, 사용자 의향, 사용 환경, 사용자 특성** 등 여러 요인이 복합적으로 영향
- 사용하는 **매체, 서비스, 시스템**에 대한 고정된 **고급 수준의 견해** (멘탈모델, 보급률 수준)에 따라 요구 사항이 달라짐

### 성능과 요구 사항의 관계
- 성능에 따라서도 요구 사항은 영향을 받음
- 자동차의 속력처럼 **연속적 값이 아닌 비이진적 데이터**(0~100점)로 평가
- 부드러운지, 빠른지, 소음 등 **파편적 관찰을 종합**해 전체 수준 판단

### 사용자 유형에 따른 차이
- **전문 사용자**와 **초보 사용자**의 요구 학습곡선이 다름
- 초보자에겐 쉽지만 전문가에겐 **편의적으로 사용하기 불편한 제품이 존재하는 이유** — 서로 다른 사용자를 대상으로 하기 때문
- **Learning Curve**를 고려한 사용성 설계 필요

## 3. 요구 사항 정의서

### 개념
- 정의 단계에서 고객의 요구를 정리해서 분석하고 종합한 각종 **요구 사항 정의서(Requirements Specification)**
- 요구 사항 정의서는 프로젝트의 **실질적인 산출물**이자 **기능 설계의 출발점**

### 구성 요소
| 항목 | 설명 |
|------|------|
| **제목** | 기능/플로우 이름 |
| **작성 사유** | 사용자 리서치에서 얻은 요구사항 근거 |
| **작성일** | 일자 기록 |
| **배경/디자인 이슈** | 문제 상황과 배경 |

### 요구 사항 정의서와 디자인 브리프의 관계
- 요구 사항 정의서는 디자인 브리프의 하위 요소로 활용
- 디자인 브리프, 프로젝트 킥오프 등의 단계에서 참조됨
- 그림 2.17과 그림 2.88을 통해 **요구 사항 정의서**의 구체적 양식 예시 제공

## 4. 사용자의 잠재 욕구 파악

### 잠재 욕구의 중요성
- 요구 사항을 결정하는 가장 중요한 요소는 바로 **사용자의 잠재 욕구를 파악**하는 것
- 고객에게 소리, 사용자의 직접적 요청 뒤에 숨은 **잠재 니즈** 발견이 핵심
- 단순히 사용자의 불만을 나열하는 것이 아니라 **근본적인 원인(Why)**을 찾아야 함

### 헨리 포드와 스티브 잡스의 사례
- 헨리 포드: "고객에게 무엇을 원하느냐 물었다면 **더 빠른 말**이라고 대답했을 것"
- 스티브 잡스: "사용자는 자기가 원하는 것을 **보여주기 전까지 모른다**"
- 이 사례들은 **고객의 요구를 그대로 수용**하는 것이 아니라, 더 나은 해결책을 제시하는 것이 디자이너의 역할임을 보여줌

### 잠재 욕구 파악의 방법
- 사용자의 의견이나 **고객의 목소리를 과정적으로 받아들이지 말고**, 근본적인 목적과 이면에 있는 **해석을 통해 새로운 관점**의 해결책 제시
- 어떤 의견(Want)을 요구로 그대로 수용하는 것보다 **한 단계 더 나아가 배경과 요구된 것을 해석하여** 잠재적 요구를 수용
- 자신만의 **이론적인 요구를 수용**하는 관점을 갖추어야 함

### 사용자의 유형 (니즈 표현 방식)
- **문제 해결** ↔ **요구** ↔ **해결/만족** (사용자의 유형별로 욕구 표현이 다름)
- 사용자의 요구보다 더 깊이 있는 **페인포인트**를 찾으면 근본적이고 혁신적인 해결책에 도달 가능

## 5. 문제 프레이밍

### 클레이튼 크리스텐슨의 관점
- **『혁신기업의 딜레마(The Innovator's Dilemma)』**의 저자 클레이튼 크리스텐슨(Clayton Christensen)
- 밀크셰이크 사례: "**슈퍼도 비교도 경쟁자도 아닌** 고객이 해결하고자 하는 과제(Job)로 봐야 한다"
- **JTBD(Jobs To Be Done)**: 고객은 제품을 구매하는 것이 아니라, **자신의 과제를 해결하기 위해 고용(Hire)**하는 것

### 프레이밍의 개념
- 문제를 **어떤 틀(Frame)**로 보느냐에 따라 해결책이 달라짐
- Before/After로 **수행해야 할 과업(찾기)** 관점에서 문제를 재정의
- 자동판매기 사례: 자동판매기를 사용하는 것은 "음료를 사는 것"이 아니라 **"갈증을 해소하거나 시간을 보내는 것"**
- 스타벅스 사례: 커피를 파는 것이 아니라 **"제3의 공간"**을 제공

### 프레이밍의 핵심
- 문제를 **올바르게 프레이밍**하면 혁신적인 해결책이 도출됨
- 두 가지를 **해체하고 다시 재구성**하면 새로운 관점의 비즈니스 모델/솔루션 가능

## 6. 문제 정의와 UX 전략의 관계

### UX 전략 도출
- 여러 리서치 데이터 분석을 통해 디자이너는 결국 **그래서 뭘 해야 하는지**, 해당 **핵심 요인은 어떻게 찾아내야 하는지** 알아내야 함
- 리서치 결과를 **해결책을 찾는 과정(Finding)**으로 연결하고, 정의 단계와 거기에 뒤따르는 인사이트로 **문제를 해결(Solving)** — 개발 및 단계의 산출물이 됨

### 디자인 문제 발견 → 디자인 문제 해결
| 디자인 문제 발견 | 디자인 문제 해결 |
|------|------|
| 요구사항/근거 → 페인포인트 → UX 전략 → 아이디어 | 사업 전략 & 해결 방향 단계 제시 |
| **기획 요건** 포함 | **주요 시간 산정 & 단계 계획** |

### 페인포인트와 기회 요건
- **페인포인트 & 니즈**: 구체적 사용자 불편/문제 나열
- **기획 요건**: 페인포인트에서 도출된 해결 방향과 전략
- 페인포인트에서 니즈로, 기획 요건에서 시스템과 기능에 대한 구체적 전략 도출

### 실무 적용
- 벽면에 포스트잇으로 화이트보드에 페인포인트를 정리하고 아이디어를 발산
- **승인권자와의 커뮤니케이션**: 발견 내용과 해결 방향을 팀원/이해관계자와 공유
- 수렴하기보다는 **발산적 아이디어를 디바이스 되기 위한 방향** 제시

## 핵심 인사이트

1. 디자인 문제 정의는 **더블 다이아몬드의 첫 번째 수렴 단계**로, 리서치에서 무엇을 디자인할지 결정하는 핵심 과정
2. **요구 사항 ≠ 기능** — 요구를 기능으로 변환하는 과정은 해석과 판단이 필요한 디자인 행위
3. **잠재 욕구 파악**이 혁신의 핵심 — 헨리 포드와 스티브 잡스 사례처럼 고객이 말하지 않는 니즈를 발견해야 함
4. **문제 프레이밍(JTBD)**: 문제를 어떤 틀로 보느냐에 따라 해결책의 방향이 완전히 달라짐
5. 문제 정의에서 **UX 전략**으로 이어지며, 페인포인트 → 기획 요건 → 디자인 해결책의 흐름을 따름
